> **Opt-in skill.** Per the Sprint Spine invocation discipline (opt-in, not auto-fire): invoke explicitly when an idea genuinely warrants the interrogation. Default is lean judgment. Adapted from a YC-office-hours diagnostic; it pairs with — but never substitutes for — `sprint-think`.

## When to use

When a raw, unvalidated product or business idea is on the table and the operator wants it pressure-tested — "is this worth building?", "poke holes in this." It interrogates the *idea*, not a document.

Pick the right gate:
- **office-hours** → interrogate a raw idea. Output is a pursue / refine / kill verdict + a one-week experiment. It does NOT produce a Think doc, does NOT feed Build, does NOT build anything.
- **sprint-think** → frame an *already-approved deliverable* and feed the build pipeline. Use once the decision to build is made.
- **ceo-review** → the sharp prioritization / tradeoff gate on something already scoped.

The orchestrator dispatches one founder/investor specialist; it does not answer the questions itself.

## Two modes

- **Startup mode (default)** — a real venture. Run the six forcing questions. The status quo is the real competitor.
- **Builder mode** — a side project / hackathon / learning build with no commercial thesis. Lighter: surface the most exciting version, not the most optimized one. Upgrade to startup mode if the conversation turns to revenue or customers.

## Process (startup mode)

Run the diagnostic — don't interrogate mechanically; if context already answers a question, use it and ask only the highest-leverage missing one. Be anti-sycophantic: challenge vague market language, "platform" visions, and generic "AI for X" framing. A compliment or a signup is not demand — observed behavior is.

**The six forcing questions:**
1. **Demand reality** — what proves someone *wants* this? Behavior, not compliments. A top-3 pain or a vitamin?
2. **Status quo** — what does the user do today instead? That workaround is the real competitor. "Nothing" usually means the pain isn't sharp enough to act on.
3. **Desperate specificity** — name the single most desperate first user: role, situation, urgency. Who'd use a duct-taped v0 tomorrow?
4. **Narrowest wedge** — the smallest useful version someone would use or pay for this week.
5. **Observation & surprise** — what has the operator personally observed that a generic analyst would miss? The non-obvious insight the idea rests on.
6. **Future-fit** — does this become more essential in three years? What changed to make now the moment?

**Two sharpeners:**
7. **Killer assumption** — the single belief that, if false, kills the thesis. State it falsifiably.
8. **Seven-day assignment** — the cheapest concrete experiment that de-risks the killer assumption in one week.

**Optional cross-model second opinion:** for a high-stakes idea, run [codex-adversarial](../codex-adversarial/SKILL.md) as an independent advisor — steelman the strongest version, name the one most revealing answer, name one wrong agreed-premise plus the evidence that would disprove it, and a 48-hour prototype spec.

## Output

A structured business-review verdict (a few bullets — file a durable brief only if the idea is substantial):
- **Verdict** — pursue / refine / park / kill (one line + why)
- **Real user & pain** · **Status quo** · **Narrow wedge** · **Observation / why-now** · **Killer assumption** · **Seven-day assignment** · **Open questions**

Founder/investor voice: skeptical, specific, evidence-demanding.

## Gate

This skill ENDS at the verdict + assignment. Advisory only — it never produces a deliverable and never auto-chains into the spine. If the verdict is "pursue," the next step is to run the experiment; only if it passes do you frame the build via `sprint-think`.
